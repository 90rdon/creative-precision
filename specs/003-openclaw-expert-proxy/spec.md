# Feature Specification: OpenClaw Expert Assessment Proxy

**Feature Branch**: `003-openclaw-expert-proxy`  
**Created**: 2026-03-01  
**Status**: Draft  
**Input**: User description: "Connect AI assessment and synthesis on the site pages to OpenClaw Expert agent via the proxy node server."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Seamless Expert AI Assessment (Priority: P1)

As a high-intent executive, I want to land on the website, start a free assessment, and converse directly with an intelligent "Expert" AI that understands my context and dynamically synthesizes its findings, so that I receive elite-level strategic advice simply by chatting.

**Why this priority**: Core value proposition. Provides the foundational "Executive Experience" by ensuring the conversation is powered completely by the intelligent OpenClaw Expert agent via a transparent proxy layer, without any intermediate "dumb" phases.

**Independent Test**: Start a new assessment chat on the local site and verify that responses are generated by the OpenClaw expert agent seamlessly through the proxy server.

**Acceptance Scenarios**:

1. **Given** a user initiates the AI assessment, **When** they send a message, **Then** the proxy server routes the message to the OpenClaw Expert agent.
2. **Given** the OpenClaw agent processes the message, **When** it responds, **Then** the proxy server streams the generated response back to the web chat interface seamlessly.

---

### User Story 2 - Admin Lead Notification (Priority: P2)

As the Admin (9_0rdon), I want to receive a Telegram notification whenever an executive starts the assessment session so that I can monitor the lead quality and high-value conversations in real-time.

**Why this priority**: Business intelligence. Allows the human lead to stay informed about high-intent interactions on the website without being a bottleneck.

**Independent Test**: Trigger a new assessment session on the web and verify a message is sent to the Admin Telegram bot with the initial lead details/session ID.

**Acceptance Scenarios**:

1. **Given** an assessment session is initialized by a new visitor, **When** the proxy establishes the connection to OpenClaw, **Then** the Telegram channel sends a notification message to the Admin containing the new `Session ID` and relevant initialization metadata.

---

### Edge Cases

- **What happens when the OpenClaw expert agent is slow or generating a complex response?**
  - The web UI should show a thinking/typing indicator (e.g., "The Expert is analyzing...") to buffer the latency and provide visual feedback.
- **How to handle refreshing the browser mid-assessment?**
  - The session ID must be persisted in the browser's `localStorage` so the proxy can reconnect the web client to the existing OpenClaw session upon reload.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Proxy server MUST establish a persistent connection between the `assessment.html` chat interface and a dedicated OpenClaw Expert agent session.
- **FR-002**: Proxy server MUST bi-directionally stream messages: forwarding user inputs to the OpenClaw agent, and returning OpenClaw responses back to the UI.
- **FR-003**: System MUST trigger an asynchronous Telegram notification to the Admin channel upon assessment session initialization.
- **FR-004**: System MUST maintain session persistence (Mapping Browser Session -> OpenClaw Session ID) to allow for page reloads without losing the conversation context.

## Clarifications

### Session 2026-03-01
- Q: Is there a "dumb" AI phase before a "smart" expert phase? → A: No. The user interacts exclusively with the OpenClaw Expert agent from the very first message via the proxy. There is no "switch" or "handoff".
- Q: What level of detail should be included in the Admin Telegram alert? → A: Context Summary Only. Telegram is strictly for Admin alerts, not for User interaction.

### Key Entities

- **Expert Session**: The persisted OpenClaw context for a specific web user.
- **Admin Alert**: The Telegram message sent to the Admin channel.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of assessment chat messages successfully proxy to and from the OpenClaw agent.
- **SC-002**: End-to-end latency (User sends message -> Interface displays first token of response) is minimized to maintain conversational flow.
- **SC-003**: 100% of new assessment sessions trigger an Admin alert on Telegram.
